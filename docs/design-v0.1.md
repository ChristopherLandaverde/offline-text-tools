## Problem Statement

Grammar fixes and EN↔PT translations that run locally by default, triggered by
a global hotkey, with a mandatory diff review before any text is applied. The
existing Linux prototype (rofi popup, Ctrl+Shift+L) proves the loop works. v0.1
formalizes the architecture so: (a) the same core runs on Linux and macOS,
(b) local-first stays the default but hosted API providers can be plugged in,
and (c) adding a new command is a data-only change.

## What Makes This Cool

- **The review IS the product.** Every edit shows a diff; nothing auto-applies.
- **Local-first by default, hosted when you want it.** Ollama runs on your
  machine with zero config. Drop in an OpenAI / Anthropic / custom key and the
  same frontends call a hosted model instead. Your choice, per action.
- **Two popups, one brain.** rofi on Linux and `choose` on macOS both call the
  same Python core — same prompts, same safety rules, same diff output.
- **New command = new prompt file.** No code changes needed to add a fourth
  or fifth action.

## Constraints

- macOS + Linux, not Windows
- Local Ollama is the default path; hosted is opt-in
- Clipboard in / clipboard out for v0.1 (no accessibility-API replacement)
- Diff review is mandatory on every invocation
- No telemetry, no phone-home, ever

## Premises

1. Provider abstraction exists from day one. Ollama, llama.cpp, and hosted
   HTTP providers (OpenAI-compatible, Anthropic) are all subclasses of the
   same `Provider` base.
2. Clipboard-based I/O. Direct in-place text replacement is v0.2+.
3. Diff review always runs. No auto-apply path.
4. Safety rejects are hard stops. Any change to URLs, emails, or numbers
   between input and output refuses the edit.
5. Python core, thin platform frontends.
6. Actions are config-driven. `actions.toml` + `prompts/*.txt` defines every
   command. The Python core is action-agnostic.
7. API keys live in the OS keychain or a local config file — never in the repo.
   Local-first means hosted providers are off unless explicitly enabled.

### Frontend ↔ core contract

Frontend invokes:
```
python -m offline_text run <action> [--provider <name>] < input.txt
```
Core returns JSON on stdout. Two top-level shapes, discriminated by `status`:

**Success / safety-reject:**
```json
{
  "status":   "ok" | "reject",
  "action":   "fix-grammar",
  "provider": "ollama/llama3.1:8b",
  "original": "...",
  "fixed":    "...",
  "edit_ratio": 0.07,
  "safety": { "status": "ok" | "reject", "reason": "..." },
  "diff_unified": "...",
  "diff_word":    "..."
}
```

**Error (provider/network/auth failure):**
```json
{
  "status":     "error",
  "error_kind": "auth" | "network" | "rate_limit" | "model" | "unknown",
  "message":    "Human-readable message shown in the popup"
}
```

Exit codes follow sysexits.h: 0 = ok, 65 = reject, 69 = provider unavailable,
77 = auth, 75 = rate limit.
Frontend renders the diff, prompts for accept/cancel, on accept writes `fixed`
back to the clipboard.

Additional entry points:
```
python -m offline_text list-actions     # reads actions.toml
python -m offline_text list-providers   # which providers are configured
python -m offline_text doctor           # checks Ollama running, key present, etc.
```

### Action registry (extensibility point #1)

`actions.toml`:
```toml
[[action]]
id          = "fix-grammar"
label       = "Fix grammar"
prompt_file = "prompts/fix-grammar.txt"
provider    = "ollama"           # default; user can override per action
max_edit_ratio = 0.40            # triggers safety "aggressive rewrite" flag

[[action]]
id          = "translate-en-to-pt"
label       = "EN → PT"
prompt_file = "prompts/translate-en-to-pt.txt"
provider    = "ollama"

[[action]]
id          = "translate-pt-to-en"
label       = "PT → EN"
prompt_file = "prompts/translate-pt-to-en.txt"
provider    = "ollama"
```

Adding a new action (e.g., `summarize`, `formalize`, `translate-es-to-en`) is:
1. Drop a prompt file in `prompts/`
2. Add a `[[action]]` block in `actions.toml`

No Python changes. The frontend popup lists available actions by reading
`list-actions`.

### Provider abstraction (extensibility point #2) — LiteLLM-backed

Per eng review: use **LiteLLM** instead of hand-rolled httpx clients. One
`completion()` call routes to any of 100+ providers based on the model string
(`ollama/llama3.1:8b`, `gpt-4o-mini`, `anthropic/claude-haiku-4-5-20251001`,
`groq/llama-3.3-70b-versatile`, any OpenAI-compatible endpoint). Adding a new
provider is a config-line change, not a new Python file.

```python
# providers/litellm_adapter.py  — ~20 lines total for all providers
import litellm  # lazy-imported at call site, not module top, to keep CLI snappy

def run(model: str, system_prompt: str, text: str) -> str:
    resp = litellm.completion(
        model=model,
        messages=[
            {"role": "system", "content": system_prompt},
            {"role": "user",   "content": text},
        ],
    )
    return resp.choices[0].message.content

def healthcheck(model: str) -> tuple[bool, str]:
    # cheap ping: 1-token completion with 2s timeout
    ...
```

LiteLLM handles auth (reads `OPENAI_API_KEY`/`ANTHROPIC_API_KEY`/etc. from env
or explicit param), normalizes error shapes across providers, and supports
retries/backoff. Cost: ~15MB dep + 200-400ms cold import, mitigated by lazy
import.

User's `~/.config/offline-text/config.toml`:
```toml
default_provider = "ollama"

[providers.ollama]
base_url = "http://localhost:11434"
model    = "llama3.1:8b"

[providers.openai]
# api_key lives in OS keychain, not here
base_url = "https://api.openai.com/v1"   # or any compatible endpoint
model    = "gpt-4o-mini"

[providers.anthropic]
# api_key in keychain
model = "claude-haiku-4-5-20251001"
```

API keys stored via `keyring` (macOS Keychain, libsecret on Linux).
`offline-text config set-key openai` prompts for the key and stores it.
Keys are never read from config files or env vars by default. (Env-var fallback
is configurable for CI/headless setups.)

### Per-action provider override

Actions in `actions.toml` can pin a provider:
```toml
[[action]]
id = "fix-grammar"
provider = "ollama"          # stays local — fast, private, good enough

[[action]]
id = "translate-en-to-pt"
provider = "anthropic"       # user decides hosted is worth it for translation
```

Frontend can also pass `--provider` at runtime to override per-invocation.

### Safety layer

Implemented in `safety.py`, unchanged by provider. Extracts sets of URLs,
emails, and integers/decimals from `original` and `fixed`. If sets differ,
return `reject` with the specific invariant violated. Also enforces
`max_edit_ratio` from the action config.

### Diff rendering

Core produces both unified and word-level diffs. Frontends render whichever
suits the surface:
- **Linux rofi:** unified diff in a `-mesg` pane; `Accept` / `Cancel` rows.
- **macOS:** action pick via `choose`, then the diff renders in a terminal
  popup (iTerm/Terminal) via `colordiff` or `delta`, followed by a `y/n`
  accept prompt. `choose` is a list-picker, not a diff viewer — the terminal
  popup keeps the CLI aesthetic and works offline with no UI framework.
- **v0.2 upgrade path:** Raycast extension with markdown-rendered word diff.

### Clipboard handling (atomic by design)

Frontend reads the clipboard once at start; **never writes unless the user
hits Accept**. Cancel is mechanically incapable of corrupting the clipboard
because no write happens on the cancel path. No backup-and-restore logic.

## Approaches Considered

### Approach A: Pure shell, no Python

Bash + jq on both OSes. Rejected — safety regex, TOML parsing, diff,
and provider HTTP calls are all painful in bash. Testing is harder.

### Approach B: Python core + shell frontends (chosen)

Python does the work; shell wrappers handle popup + clipboard.

### Approach C: Single Tauri/Electron app cross-platform

One binary, rich UI. Rejected for v0.1 — weeks of work, and the rofi version
already works well. Stays on the roadmap.

## Recommended Approach

**Approach B.** ~300 lines of Python (up from 200 because of the hosted
providers + action registry + keyring integration), two ~40-line shell
frontends.

## Feature Roadmap (extensibility examples)

Easy adds, in rough priority order:

- **`summarize`** — TL;DR of selected text. Just a prompt file + action entry.
- **`formalize` / `casualize`** — tone shift. Same pattern.
- **`translate-es-to-en` (and more languages)** — new prompt per pair.
- **Custom actions** — user drops their own prompt file in
  `~/.config/offline-text/prompts/` and adds an entry to their personal
  `actions.toml`; merges with the shipped defaults.
- **Provider: Groq / Together / local vLLM** — already covered, they're
  OpenAI-compatible. Point `base_url` at them.
- **Provider: Claude API** — shipped in v0.1 as a second hosted option.
- **v0.2 actions:** `explain-code`, `rewrite-for-clarity`, `fix-punctuation`,
  dictation cleanup.

## Open Questions

- Default local model for EN↔PT translation quality. `llama3.1:8b` may be

- Rofi `-mesg` pane length limits for long paragraphs — fall back to

- Whether to support streaming output in the diff view (providers support it;
  makes the UX feel faster but complicates diff rendering).

## Success Criteria

- Global hotkey triggers the popup in <200ms on both OSes
- 50-word grammar fix completes in <3s end-to-end on modern hardware with
  local `llama3.1:8b`
- Safety layer catches every URL/email/number alteration in the evaluation set
- Adding a new action requires zero Python changes
- Swapping Ollama for a hosted provider requires only a config edit + keyring
  entry — no code changes
- Zero clipboard corruption on cancel

## Per-Machine Deployment

Config is per-machine (`~/.config/offline-text/config.toml`), so the same repo
runs differently on different hardware:

**Linux laptop with decent VRAM** — everything local:
```toml
default_provider = "ollama"
[providers.ollama]
model = "qwen2.5:14b"     # or larger if VRAM allows
```

**Base Mac mini (memory-constrained)** — per-action split:
```toml
[providers.ollama]
model = "llama3.2:3b"     # tiny, fast, fits easily
[providers.anthropic]
model = "claude-haiku-4-5-20251001"
```
Then in `actions.toml` override per action:
```toml
[[action]]
id = "fix-grammar"
provider = "ollama"        # 3B local is fine for grammar, stays offline
[[action]]
id = "translate-en-to-pt"
provider = "anthropic"     # quality matters, hosted is worth it
```

Rationale: grammar is a small-delta, high-frequency task — local wins on
latency and privacy. Translation is high-delta, lower-frequency — hosted
quality is worth the API cost on memory-constrained hardware.

## Linux Migration Path

Existing Linux rofi prototype is preserved through the v0.1 cutover:

1. **Before refactor:** current script is copied to
   `frontends/linux/popup-legacy.sh` and tagged `pre-v0.1-linux-prototype`
   in git. If anything regresses, the legacy script is one command away.
2. **During refactor:** new `frontends/linux/popup.sh` is written alongside,
   calling `python -m offline_text run`. The old hotkey can still point at
   the legacy script while the new one is tested.
3. **Cutover:** once the new popup matches the old flow against the eval
   set (same inputs produce same or better outputs), flip the hotkey, keep
   legacy around for ~1 week, then delete.
4. **No change to:** Ollama daemon, models in VRAM, rofi look/feel, hotkey
   binding in the WM, clipboard tools (xclip on X11, wl-copy on Wayland —
   auto-detected), or config path (`~/.config/offline-text/` via
   `platformdirs`).

## Cross-Provider Evaluation (v0.1 scope)

Quality is testable across all configured providers using the case set in
`docs/evaluation.md`. Pulled into v0.1 from the v0.2 TODO list because
"does the hosted model actually do better than local, and by how much?" is
the question that drives the per-machine config decisions.

**Command:**
```
python -m offline_text eval [--providers ollama,openai,anthropic] \
                            [--actions fix-grammar,translate-en-to-pt,...] \
                            [--output eval-results.md]
```

**What it runs:** every case in `docs/evaluation.md` × every listed action ×
every listed provider. Each case has `input`, `task`, `expected`, `notes`,
`risk` per the existing doc.

**What it reports:**

| Case | Provider | Output | Safety | Edit ratio | Pass? | Notes |
|---|---|---|---|---|---|---|
| "run grammar on correct text" | ollama/llama3.1:8b | (same) | ok | 0.00 | ✓ | unchanged as expected |
| "run grammar on correct text" | gpt-4o-mini | (same) | ok | 0.00 | ✓ | |
| "run grammar on correct text" | claude-haiku-4-5 | (same) | ok | 0.00 | ✓ | |
| "EN→PT with embedded phone number" | ollama/llama3.1:8b | "...555-1234" | ok | 0.14 | ✓ | number preserved |
| "EN→PT with embedded phone number" | gpt-4o-mini | "...cinco-um-2-3-4" | **reject** | 0.31 | ✓ | safety caught hallucination |
| ... | | | | | | |

**Pass criteria** (from `docs/evaluation.md`): meaning preserved, grammar
improved (or unchanged when correct), tone preserved, minimal rewrite, no
entity drift. Programmatic checks catch the mechanical ones (unchanged-when-
correct, entity preservation, edit ratio). The qualitative ones (meaning,
tone) get flagged for manual review with a diff pair.

**Safety-feature coverage** is part of the eval set — each provider is run
against cases where the model is known to hallucinate numbers, URLs, or
emails. A "pass" means the safety layer rejected the bad output, not that
the model behaved well. This tests the feature we built, not just the model.

**New-feature coverage:** every feature added during this design iteration
(LiteLLM provider routing, per-action provider overrides, atomic clipboard,
error-kind schema, doctor healthcheck) has a corresponding eval or integration
test. Specifically:
- Per-action provider override → case asserts that `translate-en-to-pt` with
  `provider=anthropic` actually hits Anthropic (LiteLLM call log inspected).
- Error-kind schema → case simulates auth failure, network timeout, rate
  limit; asserts core output matches the discriminated schema.
- Doctor healthcheck → case runs with Ollama stopped; asserts exit code 69
  and "start Ollama" message.

**Cost note:** running the full eval against hosted providers costs tokens.
Typical: ~30 cases × ~200 tokens each × 2 hosted providers ≈ 12k tokens
per run. At Haiku/gpt-4o-mini prices, about $0.05 per full run. Run it
manually, not on every commit.

## Distribution Plan

v0.1 is a git-clone-and-install project:

```
git clone && cd offline-raycast
make install
```

- **Linux:** `make install` drops `offline-text` into `~/.local/bin`, installs

  `choose` popup script, prints a Raycast Script Command snippet.
